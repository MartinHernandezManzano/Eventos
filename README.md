El quinto proyecto, Eventos, cubre los ejercicios 15, 16, 17 y 18 y profundiza en el manejo de eventos. El concepto central es ActionEvent — el evento que se dispara cada vez que el usuario pulsa un botón. Ya lo habíamos usado en proyectos anteriores, pero aquí se trabaja de forma más explícita y con casos más variados.
EventoConsola resuelve el ejercicio 15 y es el más básico del bloque — un botón que imprime un mensaje en consola al pulsarlo. El objetivo es entender bien la mecánica: setOnAction() registra el manejador del evento, y la lambda que recibe es el código que se ejecuta cada vez que ocurre el clic. El parámetro event es el ActionEvent en sí, que aquí no usamos pero hay que declararlo.
EventoValidacion resuelve el ejercicio 16 y retoma la validación de formularios del proyecto anterior, pero con una novedad importante: en vez de mostrar el error en un Label dentro de la ventana, se usa un Alert — una ventana emergente. Alert tiene distintos tipos según la situación: ERROR para errores, INFORMATION para confirmaciones. El método showAndWait() muestra el diálogo y bloquea la ejecución hasta que el usuario lo cierra.
EventoEtiqueta resuelve el ejercicio 17 y va un paso más allá del simple cambio de texto — el botón alterna entre dos estados cada vez que se pulsa, cambiando tanto el texto como el color de la etiqueta. Para conseguirlo se usa un boolean[] en lugar de un boolean simple, porque las lambdas en Java no pueden modificar variables locales directamente, pero sí pueden modificar el contenido de un array.
EventoMultiple resuelve el ejercicio 18 y demuestra que cada botón puede tener su propio setOnAction() completamente independiente. Sin novedades técnicas — el objetivo es ver que en una misma interfaz pueden convivir tantos manejadores de eventos como elementos interactivos haya.
